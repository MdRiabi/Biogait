Prompt



Tu es un ingénieur senior full-stack & ML spécialisé en biométrie comportementale.

Tu vas m'aider à implémenter de A à Z un système de reconnaissance biométrique par

démarche (Gait Recognition) nommé BioGait, conforme RGPD & AI Act.



=== STACK TECHNIQUE FIXÉE ===

- Backend : Python FastAPI (async), WebSockets, REST

- IA Core : MediaPipe Pose (17 keypoints), FAISS (vector search), TensorFlow/PyTorch

- Frontend : NiceGUI + TailwindCSS + Chart.js

- Mobile : PWA React + WebRTC (flux caméra smartphone → serveur PC)

- BDD : PostgreSQL (profils/métadonnées) + FAISS index (vecteurs biométriques) + Redis (sessions/alertes)

- Sécurité : AES-256 at rest, TLS in transit, JWT auth, RBAC

- DevOps : Docker Compose, GitHub



=== PIPELINE IA (ordre strict) ===

1. Capture vidéo → 30 FPS

2. MediaPipe Pose → extraction 17 keypoints (x, y) par frame

3. Buffer 30 frames → séquence temporelle (1 seconde de marche)

4. Feature engineering → vecteur 128D (angles articulaires, vitesse, longueur de foulée,

   rythme, oscillation latérale)

5. FAISS similarity search → score de confiance [0-100%]

6. Decision logic → seuil configurable par zone (75% zone normale / 95% salle serveurs)

7. Output → log anonymisé + push WebSocket dashboard + alerte PWA



=== 4 MODULES MÉTIER ===

Module A — Enrôlement :

  - Capture 5 séquences (angles différents, vitesses différentes)

  - Extraction + moyenne des vecteurs → profil stable

  - Stockage : vecteur chiffré AES-256 en BDD, jamais la vidéo brute



Module B — Reconnaissance temps réel :

  - Stream continu depuis caméra IP ou mobile via WebRTC

  - Identification < 2s après 30 frames capturées

  - Sortie : nom + score + zone autorisée ou ALERTE



Module C — Dashboard Admin :

  - Accès réussis (qui, quand, score de confiance)

  - Alertes actives (inconnus, accès refusés, tentatives répétées)

  - Stats : taux de reconnaissance, FAR, FRR, courbe ROC

  - Replay vidéo floutée (anonymisation RGPD)

  - Logs d'audit complets (export PDF signé numériquement)



Module D — Conformité & Gouvernance :

  - Purge automatique vidéo brute après X heures (configurable)

  - Registre de traitement ISO 27701 (finalité, base juridique, durée de rétention)

  - DPIA simulée (Art. 35 RGPD)

  - Classification AI Act (système haut risque → Art. 9, 10, 11, 12, 14, 15)



=== MÉTRIQUES DE PERFORMANCE CIBLES ===

- Accuracy ≥ 90%

- FAR (False Acceptance Rate) ≤ 5%

- FRR (False Rejection Rate) ≤ 10%

- Latence décision ≤ 2 secondes

- Dérive de démarche → déclenchement re-enrôlement automatique



=== DATASET ===

Utilise un dataset public existant (CASIA-B ou CMU MoBo ou OU-ISIR).

Adapte le preprocessing pour correspondre au format MediaPipe 17 keypoints.

=== ORDRE D'IMPLÉMENTATION (STRICT) ===



ÉTAPE 1 — Infrastructure de base

  → Docker Compose : PostgreSQL + Redis + FastAPI skeleton

  → JWT auth + RBAC (rôles : admin, superviseur, viewer)

  → Audit log middleware (chaque requête loguée avec timestamp + user + action)



ÉTAPE 2 — Pipeline IA (cœur du système)

  → Script de preprocessing dataset (CASIA-B → keypoints MediaPipe format)

  → Extracteur de features (angles, vitesse, foulée → vecteur 128D)

  → Construction index FAISS + benchmark vitesse de recherche

  → Tests unitaires : FAR/FRR sur dataset de validation



ÉTAPE 3 — Module Enrôlement

  → Endpoint POST /enroll (upload vidéo ou stream)

  → Validation : 5 séquences minimum, qualité skeleton > seuil

  → Chiffrement AES-256 du vecteur avant stockage

  → Interface NiceGUI : formulaire + visualisation skeleton temps réel



ÉTAPE 4 — Module Reconnaissance temps réel

  → WebRTC receiver → frame extraction → pipeline IA

  → WebSocket push : résultat de reconnaissance vers dashboard

  → Logique de décision par zone (seuil configurable par caméra)

  → Anonymisation automatique : flou sur visage + suppression vidéo brute après TTL



ÉTAPE 5 — Dashboard Admin

  → Vue monitoring : flux caméras actives + événements en temps réel

  → Vue alertes : fil d'alertes filtrables par niveau (CRITIQUE/IMPORTANT/INFO)

  → Vue stats : Chart.js (courbe ROC, histogramme des scores, heatmap horaire)

  → Vue logs : export PDF signé (reportlab) avec hash SHA-256 pour intégrité



ÉTAPE 6 — PWA Mobile

  → React PWA avec service worker

  → WebRTC : caméra smartphone → stream vers serveur

  → Push notifications : alertes temps réel avec photo + localisation + boutons action



ÉTAPE 7 — Conformité RGPD & AI Act

  → Module purge automatique (cron job : suppression vidéo brute après TTL)

  → Registre de traitement généré dynamiquement depuis BDD

  → Interface DPIA (formulaire structuré Art. 35 RGPD)

  → Interface supervision humaine (Art. 14 AI Act) : override manuel, justification obligatoire



ÉTAPE 8 — Tests & Évaluation

  → Mode simulation : injection de vidéos préenregistrées

  → Test attaques simples (replay attack, adversarial walking)

  → Analyse biais : performances par genre/âge/morphologie sur dataset

  → Rapport de performance final (accuracy, FAR, FRR, latence P95)



=== CONTRAINTES NON NÉGOCIABLES ===

- JAMAIS stocker la vidéo brute au-delà du TTL configuré

- JAMAIS stocker des images de visages

- Vecteurs biométriques TOUJOURS chiffrés AES-256 au repos

- TOUTE action admin loguée avec user + timestamp + hash d'intégrité

- Chaque endpoint FastAPI DOIT avoir : auth JWT + rate limiting + input validation Pydantic
